# 版本管理规范

本项目以后同时维护四条版本轴。它们用途不同，不能混用。

## 四条版本轴

| 版本轴 | 命名例子 | 作用 | 升级条件 |
| --- | --- | --- | --- |
| Git release | `v1.0.0`、`v1.1.0` | 仓库可恢复快照 | 测试通过、文档同步、必要产物不缺失后打 tag |
| 运行策略 | `1.0-cold`、`1.1-machine-registry` | Vast 启动和选机策略 | 只改变租机、WarmStart、缓存探测策略 |
| 工作流分支 | `wan_2_2_animate`、`wan_2_2_animate_segmented` | ComfyUI / RunComfy 业务链路 | 输入输出契约或生成方式变化 |
| 脚本实现 | `segmented v1/v2/v3` | 同一工作流分支下的实现迭代 | 跑通验证后才能晋升为 profile 默认入口 |

## 状态定义

- `production`：默认推荐给真实付费任务使用。
- `supported`：可以使用，但需要明确场景和参数。
- `candidate`：已跑过，但还没成为默认主线。
- `experimental`：实验入口，必须带 job name 区分，不能覆盖主线记忆。
- `deprecated`：不再使用，只保留历史恢复价值。

## 晋升规则

一个实验脚本要晋升为正式版本，必须同时满足：

1. 写入 `config/version-manifest.json`。
2. 如果成为默认入口，更新 `config/vast-workflow-profiles.json`。
3. 在 `版本.md` 的 `Unreleased` 里记录变化。
4. 至少完成一次可复现运行，产出本地文件和 R2 归档。
5. 更新 `skills/okskills/SKILL.md` 或新增对应 workflow skill。
6. 运行测试并记录结果。

## Release 规则

Git release 使用语义化版本：

- `vMAJOR.MINOR.PATCH`
- `MAJOR`：生产链路或数据契约不兼容变化。
- `MINOR`：新增可用工作流、运行策略或重要能力。
- `PATCH`：修复、文档、测试、稳定性改进。

打 release 前必须检查：

```powershell
git status --short
git status -sb
pytest -q
```

如果存在未提交内容，不能声称 GitHub 已同步到 release 状态。

## 禁止事项

- 不要把 `1.0-cold` 说成 Git 版本。
- 不要把 `wan_2_2_animate_segmented` 说成运行策略版本。
- 不要把 `segmented v3` 说成生产版，除非 profile 已切换并完成验证。
- 不要把 `output/**/workflow_runtime.json` 当源 workflow 维护。
- 不要依赖 `output/` 下的临时产物作为测试 fixture；需要测试样例时放到 `tests/fixtures/`。
